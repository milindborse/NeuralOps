System Philosophy

Not 30 bots. One intelligent ecosystem.

🧠

Central Orchestrator

Every action routes through a single brain — the Central Orchestrator Agent. It owns memory, routing, retries, conflict resolution, and escalation logic. No agent acts alone.

🔁

Retry & Resilience

Failed agent outputs trigger automatic retry with exponential backoff. After 3 failures, confidence scores drop and human-in-the-loop kicks in. The system never silently fails.

📊

Confidence Scoring

Every agent output carries a confidence score (0–100). Scores below 60 trigger verification loops. Scores below 40 force human review before any customer-facing action.

🛡️

Agent Boundaries

Each agent has hard ownership limits. No agent can override another's primary domain. Conflicts are flagged to the Orchestrator, never resolved unilaterally.

👤

Human-in-the-Loop

4 defined human intervention points: high-conflict escalation, payment anomalies, identity-challenged handoffs, and compliance-sensitive decisions. Humans override always.

🔗

Shared Memory Layer

All agents read/write to a single Customer Context Object. No agent operates on stale data. Context is versioned, timestamped, and auditable.

Core System

The Central Orchestrator — System Brain

01 Routing Engine

Every inbound event (call, WhatsApp, web form) hits the Orchestrator first. It reads the Customer Context Object, determines caller type, intent, and lead score, then routes to the correct agent chain.

Inbound Event → Context Load → Intent Classify → Route Decision
02 Memory Management

Maintains the Customer Context Object (CCO) — a versioned JSON state containing all interaction history, lead scores, booking status, and agent outputs. Every agent reads from and writes to the CCO atomically.

CCO: {
  id: "cust_8821",
  intent: "birthday_booking",
  lead_score: 78,
  conf_score: 85,
  history: [...],
  last_agent: "Lead_Qualify"
}
03 Retry & Failure Logic

Each agent invocation is wrapped in a retry envelope. On timeout or failure, the Orchestrator retries up to 3 times with exponential backoff. After 3 failures, it routes to fallback agent or human queue.

04 Confidence Score Engine

Every agent output carries a confidence score. The Orchestrator enforces thresholds before allowing downstream actions. Low-confidence outputs are quarantined until verified.

≥80: Auto-proceed
60–79: Verify loop
<60: Human review
05 Escalation Matrix

Four defined escalation triggers route from AI to human. Each has its own SLA timer and notification protocol. Humans can override, pause, or redirect any workflow in real time.

🔴Conflict score > 80 → Senior Agent
🟡Payment anomaly → Finance Review
🟡AI identity challenged → Human takeover
🔴3x retry failure → Supervisor queue
06 Conflict Resolution

When two agents output conflicting decisions (e.g., FOMO Agent says "limited slots" but Booking Agent says "available"), the Orchestrator holds the conflict, queries both with a verification prompt, and uses the higher-confidence output.

Key Tradeoffs & Decisions

Custom Orchestrator vs Off-the-Shelf

Custom build chosen because no off-the-shelf tool supports the exact retry + confidence + memory pattern needed. LangGraph or CrewAI can scaffold but need heavy customization anyway.

Dynamic Pricing Last

Pricing changes affect conversion. It's safer to optimize pricing only after we have real booking data and seasonality patterns — not theoretical models.

WhatsApp Before Email

For Indian entertainment venues, WhatsApp has dramatically higher open rates (95%+) vs email. Prioritize async WhatsApp flows in Phase 2.

Agent Governance

Ownership Boundaries & Constraints

🚫 Hard Stops — Never Cross
  • No agent can finalize a refund > ₹500 without human approval
  • AI Identity Monitor cannot lie or deny being AI after direct questioning
  • FOMO Agent cannot fabricate inventory data ("Only 2 slots" must be real)
  • Payment Agent cannot retry a failed payment without customer re-consent
  • Upsell Agent cannot add items to cart without explicit customer confirmation
  • No agent can override a human operator's active session
⚠️ Ownership Boundaries
  • Lead Qualification → owns lead scoring exclusively (Conversion Agent reads, not writes)
  • Booking Agent → sole authority on slot reservation (FOMO reads inventory, never writes)
  • Payment Validation → sole authority on transaction status
  • Conflict Predictor → can flag escalation but cannot initiate human handoff directly (must go via Orchestrator)
  • Reporting Agent → read-only access to all data, zero write permissions
🔁 Overlap Management
  • FOMO Agent + Upsell Agent: Orchestrator prevents both from triggering simultaneously (anti-spam gate)
  • Conflict Resolution + Supervisor Escalation: Resolution gets first 2 attempts; Supervisor is Level 2
  • Follow-Up + Lead Nurture: mutually exclusive triggers based on booking status flag in CCO
  • Style Adaptation + Communication: runs as a wrapper layer, not a competing agent
